# 업무 문서 작성 규칙

이 문서는 `work` 폴더 내 문서를 작성하거나 수정할 때 따라야 할 규칙을 정의합니다.

---

## 1. 문서 유형

### 1.1 프로젝트 기록 (projects/)

프로젝트 단위로 폴더를 생성하고, 그 안에 관련 문서를 작성합니다.

- **의사결정 기록** - 왜 이 기술/방법을 선택했는가
- **설계 문서** - 아키텍처, API 설계, 데이터 모델
- **회고** - 프로젝트 완료 후 배운 점, 개선점

---

## 2. 작성 스타일

### 2.1 문체

- 모든 문서는 **"~니다" 체의 공식 문어체**로 작성합니다.
- 구어체, 반말, 이모지는 사용하지 않습니다.

### 2.2 용어 사용

- **한글 우선 원칙**: 의미 전달에 불필요한 영어 단어의 괄호 병기는 지양합니다.
- **영문 병기 기준**: 최초 등장 시 개념 정의가 필요한 경우에만 `한글(English)` 형식으로 병기합니다.
- 용어 정의는 [[GLOSSARY]]를 참조하고, 새 용어는 용어집에 추가합니다.

### 2.3 명확성

- 의사결정의 배경과 근거를 명확히 기록합니다.
- "왜"를 빠뜨리지 않습니다. 결과뿐 아니라 맥락을 함께 남깁니다.

---

## 3. 메타데이터 (YAML Frontmatter)

```yaml
---
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [태그1, 태그2]
status: draft | review | done
---
```

### 3.1 태그 분류 체계

**유형 태그**
- `decision` - 의사결정 기록
- `design` - 설계 문서
- `retrospective` - 회고

**주제 태그**
- 프로젝트별 태그: 프로젝트명을 태그로 사용 (예: `auth-system`, `payment`)
- 기술 태그: `backend`, `frontend`, `infra`, `database`

---

## 4. 링크 규칙

- 업무 문서에서 사용한 기술 개념은 `learning/` 영역의 문서로 링크합니다.
- 동일 프로젝트 내 문서끼리도 상호 링크합니다.

```markdown
## 관련 문서

- [[SOLID]] - 이번 리팩토링에서 적용한 설계 원칙
- [[인증 시스템 설계]] - 이 의사결정이 반영된 설계 문서
```

---

## 5. 문서 작성 워크플로우

1. **프로젝트 폴더 확인**: `projects/` 아래 해당 프로젝트 폴더가 있는지 확인합니다.
2. **프론트매터 작성**: 날짜, 태그, 상태를 설정합니다.
3. **내용 작성**: 맥락과 근거를 포함하여 작성합니다.
4. **링크 연결**: learning 영역의 관련 문서와 링크합니다.
5. **INDEX 업데이트**: [[INDEX]]에 새 문서를 등록합니다.
